One person. One engine. Built in the open.

Kronroe is an embedded bi-temporal graph database for AI agent memory and mobile/edge applications. It's built by one person — me, Rebekah Cole — as the database engine that on-device AI memory actually needs.

↳ not an app-layer patch on something that wasn't built for it

Why I'm building it

I started Kronroe because I needed it for Kindly Roe, my iOS app. Kindly Roe is a private, on-device companion that remembers context across conversations — and it needs memory that understands time. Not just "is this fact true?" but "when was it true, when did we record it, and what did we believe before that?"

I tried to build it on top of existing tools first. Vector databases with timestamps glued on. Graph databases requiring a server. Cloud services that wanted my users' data. Each one made the same trade-off: treat temporal evolution as an application concern, not an engine primitive. Each one broke under the weight of real conversational memory after a few weeks.

The more I built, the more I realised the problem deserved a proper engine, not an app-layer patch.

So I started writing it myself. In Rust, because I wanted it to run on iOS without a runtime. With a bi-temporal model from the bottom up, because that's what the problem actually requires. Open source, because the only way embedded databases earn trust is by being inspectable.

What it is, in one paragraph

Kronroe is the DuckDB of graph databases. Like DuckDB redesigned the embedded relational engine for analytical workloads, Kronroe redesigns the embedded graph engine for temporal knowledge evolution. It runs in your process — no server, no cloud, no data leaves your device. Every fact carries four timestamps (valid_from, valid_to, recorded_at, expired_at) so you can ask not just what's true? but what was true when? and what did we believe at the time?

It ships as:

  • A Rust crate (kronroe) for native applications
  • A Python package (kronroe) via PyO3
  • An iOS framework (KronroeFFI.xcframework) for Swift apps
  • An Android library (JNI + Kotlin) for native Android
  • A WebAssembly bundle for browser apps
  • An MCP server with 11 tools for AI agents

All from one codebase, one engine, one set of guarantees. No backend to provision, no cloud account to create, no API key to manage.

Who I am

I'm Rebekah Cole. I'm a software engineer based in the United Kingdom. I'm the sole maintainer of Kronroe and the founder of Kindly Roe. I write code in Rust, Swift, Python, and TypeScript depending on the problem.

I've spent the last few years on the intersection of AI agents, on-device privacy, and personal tools. The thing I keep coming back to is that the genuinely useful AI applications — the ones that remember you, that work without an internet connection, that don't require trusting someone else's cloud — share a common bottleneck: the storage layer doesn't take time seriously. Kronroe is my answer to that.

I built this in the open because I think it's the only way embedded databases earn trust. The license is AGPL-3.0 with a commercial option, so individual developers can use it for free in open-source projects, and companies who don't want to be bound by AGPL can buy a commercial licence.

The core engine, MCP server, and platform bindings are all working. The full-text search (Kronroe's own lexical engine), vector index (flat cosine), and contradiction detection are shipping. Hybrid retrieval and the uncertainty model are behind feature flags while I tune them.

The first user is Kronroe itself, built into Kindly Roe. The second will be you, if you want to try it.
